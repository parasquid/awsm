# Browser Extension UI Design

The extension uses a modern-neutral archival-tool aesthetic: white and cool-gray surfaces, dark text, blue primary actions and focus rings, compact system typography, restrained borders, and minimal shadow. Its identity mark is a deep-navy archive box containing a warm-amber bookmark: a compact visual shorthand for protected, preserved knowledge. Chrome uses the transparent mark at 16, 32, 48, and 128 px, including the toolbar action; the silhouette and bookmark remain distinct at 16 px without text or fine strokes. It uses no UI framework. Network access is optional, limited to the exact user-approved Coordination Server origin, and carries only Account authentication material and opaque encrypted synchronization data.

The popup is a 360 px task surface with explicit onboarding, locked, capturing, success, warning, and failure states. First use makes local-only operation the primary action and presents synchronization setup secondarily. Account setup moves to a stable full-tab surface before Chrome requests server-origin permission. The setup tab owns server choice, permission, Account creation, and Vault selection as one uninterrupted journey; custom-server configuration remains in progressive disclosure there. If Account creation reports that the address is unavailable, the tab explains that the Account may already exist and offers an in-place sign-in mode without discarding the selected server or email. Capture continues in the background if the popup closes. Reopening the popup derives its state from the persisted Runtime job. A successful recent Capture remains visible for eight seconds; a vertical progress rail fills downward to communicate when the preview will dismiss and pauses while the preview is hovered or keyboard-focused.

The library is a full extension page. Vault switching and creation are secondary management tasks inside Settings rather than primary popup or Library actions. The list contains title, source URL, capture time, screenshot availability, and warnings. Detail content is decrypted only after unlock. The full screenshot is retrieved automatically and rendered as the first content preview above the technical Artifact list through a short-lived Blob URL. Text and structured content expand directly beneath their selected Artifact row. Thumbnail has no detail action. The mandatory `PRIMARY` role is labeled **MHTML** in the interface and is the only downloadable Capture Artifact. Its verified plaintext is streamed through the Chrome Download Host using a temporary extension-owned file, declared as `multipart/related`, and saved with the canonical `.mhtml` filename; it is never assigned to a frame, object, embed, window location, or executable document surface.

Vault-level maintenance follows the Library content in a lightly elevated neutral panel. Reversible storage relief is described as reducing **device storage**, never as receiving “free” storage. Permanent Vault Vacuum remains visually separated and uses the destructive treatment. Remote-only Artifacts retain their supported semantic actions and show that they are stored on the server and retrieved when opened. Screenshot retrieval, text inspection, and MHTML download surface loading, retry, success, and failure states without moving unrelated Library content.

Account settings distinguish Sign out from Reset this device. Sign out preserves local Vaults. Reset this device is a separate destructive section and typed-confirmation dialog that deletes all extension-local databases, keys, Vaults, captures, settings, and cached files while explicitly stating that server-side Account and Vault data are untouched.

Library list/detail navigation is reflected in the extension-page URL. Opening a Capture writes its `bundleId`; returning to the list removes it. Live-state reconciliation always follows the current URL so focus, synchronization, or Artifact availability invalidations cannot bounce an open detail view back to the list. While the user has a non-collapsed text selection, non-security detail rerenders wait until the selection collapses; lock and Vault-context changes still discard decrypted content immediately.

All controls are keyboard reachable, use native semantics, retain visible focus, and report asynchronous state through polite live regions. Color is never the only status signal. Motion is nonessential and reduced-motion preferences are honored.

Action layout follows three reusable patterns. A task surface uses a full-width action stack with the primary action first and secondary navigation beneath it. A compact toolbar keeps content-width controls in a wrapping row. A dialog footer stays at the bottom of the modal, places the secondary action before the primary action with the primary action at the trailing edge, and stacks both controls at full width on narrow screens. Destructive actions retain the same geometry but use the danger treatment and remain visually separated from routine controls.

Dialogs have a fixed header, a scrollable body, and a fixed footer so their title, close control, and final actions remain available in constrained viewports. The close control is always explicit and is distinct from workflow cancellation. Settings status is presented as a compact labeled summary with human-readable values. Checkbox and radio rows use an 18 px native control aligned with the first line of their label. Notices share neutral, success, warning, and error geometry: a restrained tinted surface, semantic leading border, compact padding, and live-region semantics when state changes asynchronously.
